# GitHub CSV Push — Automated HA Report Sync

> Nightly upload of all Home Assistant CSV report files to the
> `Residential-HVAC-Performance-Baseline-` repository using the GitHub
> REST API — no `git` binary required, no end-of-day race conditions.

---

## Overview

`push_csv_to_github.py` pushes every CSV report generated by the Home
Assistant HVAC monitoring configuration to the
`wkcollis1-eng/Residential-HVAC-Performance-Baseline-` repository under
`homeassistant/reports/`.  It runs at **02:00 AM** daily via a dedicated HA
automation — safely isolated from the 23:57–23:59 end-of-day CSV write
window.

The script uses only Python standard-library modules (`urllib`, `base64`,
`json`, `pathlib`) and calls the GitHub Contents API directly.  No `git`
binary, no SSH keys, no Docker sidecar, no external Python packages.

---

## Repository Layout

HA CSV exports land under `homeassistant/reports/` — deliberately separated
from the curated historical datasets in `data/`.

```
Residential-HVAC-Performance-Baseline-/
├── data/                               ← manually curated, validated historical data
│   ├── daily_temperature.csv           ← DO NOT overwrite (different schema)
│   ├── monthly_summary.csv             ← DO NOT overwrite (different schema)
│   ├── monthly_gas_scg.csv             ← billing-cycle-precise utility records
│   ├── monthly_electricity_eversource.csv
│   ├── monthly_dhw_navien.csv
│   └── monthly_hvac_runtime.csv
│
└── homeassistant/
    ├── packages/
    ├── reports/                        ← HA live CSV exports (written by this script)
    │   ├── hvac_daily_YYYY.csv         ← daily HVAC row (nightly)
    │   ├── hvac_monthly.csv            ← HVAC monthly summary (end of month)
    │   ├── hvac_setback_log.csv        ← setback events (event-driven)
    │   ├── input_number_backup.csv     ← input_number snapshot (weekly)
    │   └── utility_monthly.csv         ← utility billing + runtime (end of month)
    └── README.md
```

The `data/` files are the validated multi-year baseline datasets with
billing-cycle-precise dates.  The `homeassistant/reports/` files are
calendar-month HA exports with their own schemas — they provide a real-time
data trail and complement (not replace) the `data/` files.

## `utility_monthly.csv` Schema

Calendar-month snapshot of all utility billing and HVAC runtime data
available in HA.  Complements the billing-cycle-precise `data/` files.

| Column | Source entity | Unit | Notes |
|--------|--------------|------|-------|
| `month` | `now().strftime('%Y-%m')` | YYYY-MM | Calendar month |
| `gas_ccf` | `input_number.gas_bill_ccf` | CCF | Total gas this bill |
| `gas_cost` | `input_number.gas_bill_amount` | $ | Total gas charges |
| `gas_days` | `input_number.gas_bill_days` | days | Billing period length |
| `elec_kwh` | `input_number.electricity_bill_kwh` | kWh | Total electricity this bill |
| `elec_cost` | `input_number.electricity_bill_amount` | $ | Total electricity charges |
| `elec_days` | `input_number.electricity_bill_days` | days | Billing period length |
| `dhw_thm` | `input_number.dhw_bill_thm` | Thm | Navien NaviLink reading, prior month |
| `hdd65` | `input_number.hdd_cumulative_month_auto` | °F-days | Month HDD accumulator |
| `cdd65` | `input_number.cdd_cumulative_month_auto` | °F-days | Month CDD accumulator |
| `furnace_hrs` | `sensor.hvac_furnace_runtime_month_2` | hrs | Total furnace runtime |
| `heat_1f_hrs` | `input_number.zone_1f_runtime_month_acc` | hrs | 1F heating zone runtime |
| `heat_2f_hrs` | `input_number.zone_2f_runtime_month_acc` | hrs | 2F heating zone runtime |
| `cool_1f_hrs` | *(stub — 0)* | hrs | **Not yet implemented** — see below |
| `cool_2f_hrs` | *(stub — 0)* | hrs | **Not yet implemented** — see below |
| `heating_eff_ccf_per_1k_hdd` | `sensor.hvac_heating_efficiency_mtd` | CCF/1kHDD | Month-to-date heating intensity |
| `efficiency_deviation_pct` | `sensor.efficiency_deviation_month` | % | vs. expected runtime |

### Cooling columns (future work)

`cool_1f_hrs` and `cool_2f_hrs` are hardcoded to `0`.  The current HA
configuration has no `call_for_cool` binary sensors and no per-zone cooling
runtime accumulators — only CDD accumulates, not equipment runtime.

When cooling runtime sensors are added, replace the `0` literals in
`appendutilitymonthlycsv` with:
```yaml
"cool_1f_hrs": {{ states('sensor.hvac_1f_cool_runtime_month') | float(0) }},
"cool_2f_hrs": {{ states('sensor.hvac_2f_cool_runtime_month') | float(0) }},
```
The schema is already stable; no column additions or CSV migrations needed.

### Relationship to `data/` billing files

| `utility_monthly.csv` field | Nearest `data/` equivalent | Key difference |
|---|---|---|
| `gas_ccf` / `gas_cost` / `gas_days` | `monthly_gas_scg.Gas_CCF` / `Total_Charges` / `Billing_Days` | `data/` uses actual `Bill_Date`; HA uses calendar month |
| `elec_kwh` / `elec_cost` / `elec_days` | `monthly_electricity_eversource.Usage_kWh` / `Charge` / `Days` | Same billing-cycle vs. calendar-month distinction |
| `dhw_thm` | `monthly_dhw_navien.Gas_CCF` | Navien meter reads in therms; 1 thm ≈ 1.02 CCF for natural gas |
| `heat_1f_hrs` / `heat_2f_hrs` | `monthly_summary.Heat_1F_Hrs` / `Heat_2F_Hrs` | Direct equivalents |
| `hdd65` | `monthly_summary.HDD65` | Same base; minor rounding differences possible |

## End-of-Month Timing

```
23:55:00  EOD capture sequence begins
23:56:30  HDD/CDD accumulators captured (capture_daily_monthly_tracking)
23:57:00  appenddailycsv
23:58:30  appendmonthlycsv           (last day of month only)
23:59:00  appendutilitymonthlycsv    (last day of month only)  ← 30 s margin
─────────────────────────────────────────────────────────────
02:00:00  push_csv_to_github         (all five files → GitHub)
```

All five files are consistently written before the 02:00 AM push.

## Setup

### 1 — Copy updated `csv_manager.py`

The `utility_monthly` schema, `cmd_append_utility_monthly()` function, and
`append_utility_monthly` argparse subcommand have been added.  Copy the
updated file into place:

```bash
cp csv_manager.py /config/scripts/csv_manager.py
```

### 2 — Add the `shell_command` entry

Inside the existing `shell_command:` block in `configuration.yaml`
(full snippet in `utility_monthly_ha_snippets.yaml`):

```yaml
  appendutilitymonthlycsv: >-
    python3 /config/scripts/csv_manager.py append_utility_monthly --data '{
      "month":                      "{{ now().strftime('%Y-%m') }}",
      "gas_ccf":                    {{ states('input_number.gas_bill_ccf')              | float(0) }},
      ...
      "cool_1f_hrs":                0,
      "cool_2f_hrs":                0,
      ...
    }'
```

### 3 — Add the automation

Paste the `append_utility_monthly_csv` block from `utility_monthly_ha_snippets.yaml`
into `automations.yaml`.  It fires at 23:59:00 on the last day of each month.

### 4 — Add `utility_monthly.csv` to the push script

Already included in `push_csv_to_github.py`:
```python
"reports/utility_monthly.csv":  "homeassistant/reports/utility_monthly.csv",
```

### 5 — Reload and test

```
Developer Tools → YAML → Reload Shell Commands
Developer Tools → YAML → Reload Automations
```

Manual test (any day — writes a row for the current month):
```
Developer Tools → Services → shell_command.appendutilitymonthlycsv
```

Verify `reports/utility_monthly.csv` exists locally, then trigger the push:
```
Developer Tools → Services → shell_command.push_csv_to_github
  token: !secret github_token
```

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| `utility_monthly.csv` not created | `csv_manager.py` not updated | Confirm `append_utility_monthly` subcommand is present |
| Row for month already exists, skipping | Manual test run earlier in month | Expected — duplicate guard is working correctly |
| `cool_1f_hrs` / `cool_2f_hrs` always 0 | Cooling sensors not yet implemented | Expected until cooling tracking is added |
| `dhw_thm` is 0 | `input_number.dhw_bill_thm` not entered yet | Enter Navien reading before month-end trigger |
| File not pushed to GitHub | See `push_csv_to_github` troubleshooting | Check HA log for `[push_csv_to_github]` entries |
